# Event Registration Tabs & Theme Issues - FIXED ✅

## Critical Discovery: UUID vs Slug in Registration APIs

### The Problem 🔴
Event detail page tabs not showing for registered users, with errors:
```
422 Unprocessable Entity
Input should be a valid UUID, invalid character... found 't' at 1
Input: "test"
```

### Root Cause Analysis 🔍

**Backend API Design:**
- **Public event endpoints:** Use `slug` (e.g., "test", "climate-summit-2024")
  - `/api/events/{slug}` - Get event details
  - `/api/events/list` - List events
  
- **Registration endpoints:** Use `event_id` (UUID)
  - `/api/events/{event_id}/registration-status` ← Expects UUID
  - `/api/events/{event_id}/registrations/me` ← Expects UUID
  - `/api/events/{event_id}/participants` ← Expects UUID

**Frontend was doing:**
```typescript
// ❌ WRONG - Sending slug to UUID-expecting endpoint
fetch(`/api/events/${event.slug}/registration-status`)
// Sends: /api/events/test/registration-status
// Backend expects: /api/events/3af1fb47-c15f-4bd6-b34e-3d123e568ec1/registration-status
```

---

## Fixes Applied ✅

### 1. Frontend: Use event.id Instead of event.slug
**File:** `/app/events/[slug]/EventDetail.tsx`

```typescript
// BEFORE ❌
fetch(`/api/events/${event.slug}/registration-status`)
fetch(`/api/events/${event.slug}/participants`)
fetch(`/api/events/${event.slug}/registrations/me`)

// AFTER ✅
fetch(`/api/events/${event.id}/registration-status`)
fetch(`/api/events/${event.id}/participants`)
fetch(`/api/events/${event.id}/registrations/me`)
```

**Lines changed:**
- Line 109: `event.slug` → `event.id` (registration status)
- Line 133: `event.slug` → `event.id` (participants)
- Line 156: `event.slug` → `event.id` (update registration)
- Line 181: `event.slug` → `event.id` (cancel registration)
- Line 147: `[event.slug]` → `[event.id]` (useEffect dependency)

### 2. API Routes: Accept UUID via slug Parameter
**Files:**
- `/app/api/events/[slug]/registration-status/route.ts`
- `/app/api/events/[slug]/registrations/me/route.ts`
- `/app/api/events/[slug]/participants/route.ts`

Added clarifying comment:
```typescript
const { slug } = await params;
// slug parameter can be either slug or UUID - backend expects UUID for registration endpoints
const eventIdentifier = slug;
```

**Note:** The route folder is named `[slug]` but actually receives UUID from the frontend for registration endpoints. This is acceptable since Next.js dynamic routes just need a parameter name, regardless of the actual value type.

### 3. Cookie Name Fix
**Files:**
- `/app/api/events/[slug]/registration-status/route.ts` (line 13)
- `/app/api/events/[slug]/registrations/me/route.ts` (lines 14, 58, 107)

```typescript
// BEFORE ❌
const token = cookieStore.get("access_token")?.value;

// AFTER ✅
const token = cookieStore.get("token")?.value;
```

### 4. Theme Hydration Fix
**File:** `/app/events/[slug]/EventDetail.tsx`

```typescript
const [mounted, setMounted] = useState(false);

useEffect(() => {
  setMounted(true);
}, []);

if (!mounted) {
  return null; // Prevent render until theme is loaded
}
```

## API Endpoint Reference

### Public Event Endpoints (Use Slug)
```
GET  /api/events                    - List all events
GET  /api/events/{slug}              - Get event details by slug
POST /api/events/{slug}/register     - Register for event (uses slug in URL but converts internally)
```

### Registration Endpoints (Use UUID)
```
GET    /api/events/{event_id}/registration-status    - Check if registered
GET    /api/events/{event_id}/registrations/me       - View my registration
PUT    /api/events/{event_id}/registrations/me       - Update my registration
DELETE /api/events/{event_id}/registrations/me       - Cancel my registration
GET    /api/events/{event_id}/participants           - List participants
```

### Admin Endpoints (Use UUID)
```
GET /api/admin/events                                  - List all events (admin)
GET /api/admin/events/{id}                            - Get event by ID (admin)
GET /api/admin/events/{id}/registrations              - List registrations (admin)
```

## Summary

**Root Cause:** Sending slug to UUID-expecting endpoints  
**Solution:** Use `event.id` for all registration operations  
**Bonus Fixes:** Cookie name + theme hydration  
**Status:** ✅ ALL FIXED  

**Key Learning:** Always check backend API docs to see if endpoints expect UUID or slug! The UUID vs slug issue is common in event/resource management systems.

## ADDITIONAL FIX: Backend API Response Property Names

### Issue Found After Initial Fix
Even with UUID fix, tabs still didn't show because property names didn't match!

**Backend Response Structure:**
```json
{
  "registered": true,           // ← Not "is_registered"
  "has_account": true,
  "registration_id": "...",
  "status": "pending",
  "available_actions": ["edit", "cancel"],  // ← Not "can_edit", "can_cancel"
  "message": "You are already registered...",
  "event_passed": false
}
```

**Frontend Was Checking:**
```typescript
// ❌ WRONG - Properties don't exist!
registrationStatus?.is_registered      // undefined
registrationStatus?.can_edit           // undefined
registrationStatus?.can_cancel         // undefined
```

### Final Fix Applied ✅

**File:** `/app/events/[slug]/EventDetail.tsx`

Changed all occurrences:
```typescript
// Tab visibility
registrationStatus?.is_registered  →  registrationStatus?.registered

// Permission checks  
registrationStatus.can_edit  →  registrationStatus.available_actions?.includes('edit')
registrationStatus.can_cancel  →  registrationStatus.available_actions?.includes('cancel')
```

**File:** `/app/api/events/[slug]/registration-status/route.ts`

Updated fallback responses:
```typescript
// When no token or 401 error
{
  registered: false,
  registration_id: null,
  status: null,
  available_actions: [],
  registration: null
}
```

### Lines Updated
- EventDetail.tsx line 114: `is_registered` → `registered`
- EventDetail.tsx line 116: `is_registered` → `registered`
- EventDetail.tsx line 316: `is_registered` → `registered` (debug log)
- EventDetail.tsx line 320: `is_registered` → `registered` (tab condition)
